An Architectural Reconstruction & Developer's Field Guide

0Prologue: What This Document Is

This is a functional reconstruction of TreemapWriter2 — a reverse-engineering of its architecture intended to make explicit what has, through many sessions of AI-assisted development, become implicit. It separates the core infrastructure (the load-bearing walls and foundation) from the accreted feature surface (the rooms, furnishings, and additions built atop that foundation over time).

The purpose is threefold. First, to give both human and AI developers a map of the system that reveals which structures are foundational and which are derivative. Second, to assess how easy it is, today, to develop a new feature that integrates well into the overall design. Third, to identify clear wins — concrete, bounded improvements that would make the architecture more legible and the development process more tractable.

This document borrows its sensibility from two disciplines. From the philosophy of science, it takes the idea that a system's health is best understood not by cataloguing its parts but by understanding its theoretical commitments — the core principles that constrain what can be built and how. From historiography, it takes the principle that the present state of a system is always the residue of a sequence of decisions, each locally rational, whose cumulative effect may not have been planned.

A note on vocabulary. Throughout this document, key architectural concepts are introduced with definitions drawn from software engineering, labeled and set apart. These are not ornamental — they are the working language that lets you describe what you see, ask for what you want, and communicate with other developers (human or AI) about the structure of your system.

IIThe Geological Strata

The most important thing to understand about TreemapWriter2 is its layered architecture. Like geological strata, each layer was deposited with a specific purpose, and the layers bear a strict dependency relationship: upper layers depend on lower ones, but never the reverse.

Layered Architecture

A design in which the system is organized into horizontal layers, each providing services to the layer above and consuming services from the layer below. The crucial discipline: dependencies point downward only. A presentation component may call a service, but a service must never import a React component.

Dependency Inversion

The principle that high-level modules should not depend on low-level modules; both should depend on abstractions. In TreemapWriter2, this is realized by two seams: the Repository interface (persistence) and the AIProvider interface (AI). Components never call IndexedDB, SQLite, or a vendor AI SDK directly — they call the interface, and the implementation is injected at module load.

These two seams are the most architecturally significant feature of the codebase. They are what allowed the app to migrate from IndexedDB to SQLite without touching a single component. They are what allows three AI providers (Gemini, Anthropic, Ollama) to coexist behind one interface. Protect these seams. Any change that bypasses them — a component calling invoke() directly, or importing @google/genai — is an architectural violation.

The Infrastructure Layer

The Rust backend (src-tauri/) owns everything that touches the filesystem, the database, git, and the OS keyring. It exposes a clean IPC surface of ~34 commands organized by concern: project_*, snapshot_*, sync_*, credentials_*, agent_*. The React side never touches disk directly.

The Repository Layer

The Repository interface (src/services/repository.ts, 336 lines) defines ~36 methods covering CRUD, snapshots, git sync, search, and sessions. Two implementations exist: browser-repository.ts (IndexedDB via idb-keyval) and tauri-repository.ts (wraps Tauri invoke() calls). The registry picks one at module load.

The Service Layer

The AIProvider interface (src/services/ai-provider.ts, 937 lines) defines ~35 methods spanning diagnostics, specs, analysis, revision, gist, sprints, comparison, doctor, atmosphere, and agent operations. Each method is implemented in a dedicated ai-provider.*.ts file under src/services/ai/ (20 files). The prompts registry (src/services/prompts/registry.ts) catalogs all 68 prompt templates.

The State Layer

Zustand manages all application state through 16 slices, combined in src/state/index.ts. The slices are partitioned by lifecycle, not by feature — a principled choice documented in AGENTS.md.

The Presentation Layer

React components live under src/features/, organized by UI region. The three-column layout (Sidebar | Editor | TestsPanel) is the default view; workspaces and modals overlay it as full-screen or dialog surfaces. Components subscribe to the Zustand store directly via useStore() selectors — no React Context, no props-drilling intermediary.

IIIThe Domain Types: A Philosophical Theory Encoded in TypeScript

The domain types in src/types/index.ts are, as VISION.md states, "the most carefully considered part of the codebase." They are not a CRUD schema. They encode a genuine theory of academic prose structure — what a section does in an argument, what moves it must make, how well it realizes its specification, and how the parts relate to the whole.

Hard Core (Lakatos)

In Imre Lakatos's philosophy of science, the hard core of a research programme is the set of theoretical commitments that practitioners treat as non-negotiable. Anomalies are deflected by modifying the protective belt of auxiliary hypotheses, never the core. In TreemapWriter2, the domain types are the hard core. The features built atop them are the protective belt.

These five layers grew in chronological order as features shipped, but they coexist by design. Each models a different aspect of the writing process:

  • Layer 1 answers: "What must this section do, and how well does it do it?"
  • Layer 2 answers: "What does this section actually argue?"
  • Layer 3 answers: "How should the prose change, and what is the evidence?"
  • Layer 4 answers: "What should I write in the next 25 minutes?"
  • Layer 5 answers: "What are the functional parts of this argument, independent of the heading structure?"

The relationship between these layers is accretive, not replacive. Layer 2 does not supersede Layer 1; it adds a different analytical instrument. This is a philosophical commitment: the argument has multiple dimensions that no single analysis can exhaust.

The Source-of-Truth Hierarchy

The most important architectural decision in the entire system: every datum has exactly one canonical home. Everything else is a cache or projection.

DataSource of TruthCache / Projection
Dissertation proseproject.md on diskSQLite sections; Zustand document-state
Section specs.twriter/specs/*.spec.yamlSQLite specs
Sources.twriter/sources.jsonZustand document-state
Provenance marks.twriter/provenance.jsonZustand document-state
History.git/Snapshot list in UI
UI stateZustand (in memory)uistate.json (per-project)
Single Source of Truth (SSOT)

The principle that every piece of data in a system has exactly one authoritative location. All other representations are derived and may be stale. When they conflict, the authoritative location wins. In TreemapWriter2, the prose on disk is canonical; the in-memory state is a projection that can always be rebuilt.

VThe Feature Tax: What a New Feature Costs

This is the section that answers the user's central question: how easy is it to develop a new feature that integrates well?

The answer is: the architecture makes it possible, but the surface area makes it expensive. The "where to put X" guide in AGENTS.md is one of the best developer-scaffolding documents in any codebase I have examined — it tells you exactly where every kind of addition goes. But the number of places you must touch for a single feature is high.

Surface Area

The total number of distinct locations in a codebase that must be modified to implement a single change. High surface area means more opportunities for error, more files to understand, and more merge conflicts. It is the transactional cost of a feature.

For a feature like the Outline Doctor (shipped 2026-07-09), this pattern required:

  • New types in types/index.ts (DoctorTask, DoctorChecklist, etc.)
  • A new state slice (state/doctor-state.ts, 12 files in the workspace)
  • A new modal flag in ui-state.ts
  • New methods on the AIProvider interface
  • Two new implementation files (ai-provider.doctor.ts, ai-provider.doctor-wizard.ts)
  • New prompt .md files + registry entries
  • A new workspace directory (features/doctor/ with 13 files)
  • New utility helpers (lib/doctor-helpers.ts)
  • Wiring in ModalLayer

This is approximately 8 integration points across 5 layers. The architecture directs you to the right places (the "where to put X" guide is excellent), but the sheer count is what makes each feature expensive. This is not a design flaw — it is a consequence of taking separation of concerns seriously in a feature-rich application. But it does mean that the architecture has a higher floor for new features than a simpler system would.

VIThe Assessment: Virtues and Tensions

A mature assessment distinguishes between what works and what creates friction. TreemapWriter2 has genuine architectural virtues — decisions that would survive a clean-room reimplementation because they solve real problems well. It also has tensions — not failures, but places where the current structure resists the work being asked of it.

Essential vs. Accidental Complexity (Brooks)

Fred Brooks's famous distinction: essential complexity is inherent in the problem (a dissertation-analysis tool that handles specs, diagnostics, revision, sprints, and structural parts is complex). Accidental complexity is complexity introduced by the solution's implementation (a 937-line god-interface, 26 modal booleans, a 771-line App.tsx). The goal is never to eliminate all complexity, but to ensure that the complexity you carry is essential, not accidental.

The Virtues (what to protect)

Architectural Virtue

The domain types encode a genuine theory

The types in src/types/index.ts are not a CRUD schema with accidental structure. They model SectionFunction (what a section does in an argument), RequiredMove (what it must accomplish), DiagnosticResult (how well it succeeds), and StructuralPart (the argument's functional parts independent of headings). This is a philosophical commitment that shapes every feature. Protect it: never collapse these types into something "simpler."

Architectural Virtue

The two dependency-inversion seams are clean and load-bearing

The Repository and AIProvider interfaces are genuine abstractions, not leaky wrappers. The app migrated from IndexedDB to SQLite without touching components. Three AI providers coexist behind one interface. This is textbook dependency inversion, and it works.

Architectural Virtue

The documentation is exceptional

VISION.md (why), AGENTS.md (how), STATUS.md (now), migration-log.md (history) form a four-document system that is among the best-maintained developer documentation in any codebase of this size. The "where to put X" table alone saves hours per session. This is agent-legibility treated as a first-class product requirement.

Architectural Virtue

State partitioned by lifecycle, not by feature

UI ephemera (modal flags, panel widths) never shares a slice with domain data (markdown, testSuite). This is a principled design that prevents the "god store" anti-pattern. Cross-slice mutations are centralized in project-state, making the coupling explicit and auditable.

Architectural Virtue

Prompts are content, not code

68 prompt .md files, each imported as raw strings, cataloged in a single registry with three-tier resolution (built-in < global user overrides < per-project overrides). The prompts are editable by the user. This is a well-designed content management system for AI instructions.

Architectural Virtue

The workspace pattern is consistent and well-factored

All 11 workspaces follow the same skeleton: a state slice, a *Workspace.tsx, a *TopBar.tsx, a use-*-actions.ts hook, and a mounting point in ModalLayer. This consistency means that once you understand one workspace, you can navigate any of them.

The Tensions (where friction accumulates)

Structural Tension

The AIProvider interface is becoming a god object

At 937 lines and ~35 methods, the AIProvider interface knows about diagnostics, specs, analysis, revision, gist, sprints, comparison, doctor, atmosphere, segmentation, structural parts, and agent operations. Every new AI-backed feature adds another method to this single interface. This is the Interface Segregation Principle violation: consumers are forced to depend on methods they don't use.

Structural Tension

App.tsx retains residual business logic

At 771 lines, App.tsx still contains: 5 export handler functions (nearly identical blob-download patterns), the diagnostic runner (handleRunTests, ~120 lines), the command palette definition (27 entries), and import handling. These are feature-level concerns living in what should be a thin layout shell. The "audit 4.2" extraction of ModalLayer was a good step; the remaining handlers are the next seam.

Structural Tension

57 modal files in a flat directory

The features/modals/ directory contains 57 files. Only sprint/ (13 files) and topo/ (11 files) have their own subdirectories. The Sprint system is a full workspace-class feature buried inside "modals"; the Topo visualization is a ~3,000-line standalone system with its own force simulation. Finding a specific modal requires knowing its name. This is a discoverability problem.

Structural Tension

No shared button or input component

The pattern font-mono text-[9px] uppercase tracking-[0.12em] appears hundreds of times across 103 files. Despite having CSS component classes (hld-btn, hld-lit), most buttons are styled inline in Tailwind. There is no shared Button, Input, or Select component. Any visual consistency change requires touching dozens of files.

Structural Tension

Two modal framing systems coexist

ModalShell (the canonical system: square corners, cyan/magenta accent line, monospace titles) is used by 17 modals. But ConfirmModal, GrimoireModal, and several others use bespoke layouts with rounded-lg, backdrop-blur-sm, and different visual treatments. This creates visual inconsistency that the user can feel even if they can't name it.

Structural Tension

The store import path migration stalled

31 files still import from the deprecated src/store re-export barrel while 77 use the current src/state path. This is a minor inconsistency but it signals: mechanical cleanup does not get prioritized, which is a symptom of the AI-assisted development pattern (each session adds features; none cleans up the last session's debris).

Technical Debt

The accumulated cost of past decisions that were locally rational but create ongoing friction. Like financial debt, it is not inherently bad — taking on debt to ship a feature can be the right call. The danger is untracked debt: decisions whose costs are invisible because no one recorded them. TreemapWriter2's STATUS.md "Lingering" section is a debt register — an unusually honest one.

God Object / God Interface

A type that knows too much — it has too many responsibilities, too many methods, or too many reasons to change. The AIProvider interface at 35 methods is approaching this threshold. The cure is not to split the implementation (which is already well-factored into 20 files) but to split the interface into smaller capability contracts that consumers can depend on selectively.

These are concrete, bounded improvements ordered by their ratio of impact to effort. Each addresses a specific tension identified above. None requires rethinking the architecture — they work with the existing structure, not against it.

VIIIThe Vocabulary: A Glossary of Architectural Concepts

This section collects the vocabulary introduced throughout the document, supplemented with additional terms that are useful for reasoning about this system's evolution.

Separation of Concerns

The principle that a module should have one reason to change. The original App.tsx had at least seven (layout, persistence, AI orchestration, export, import, command palette, autosave). Each extraction reduced its reasons to change by one.

Coupling (Tight / Loose)

How much one module needs to know about another's internals. Tight coupling: a component calls invoke('project_write', ...) directly (needs to know Tauri's IPC contract). Loose coupling: a component calls repository.setProject(data) (needs to know only the Repository interface). TreemapWriter2's dependency-inversion seams create loose coupling at the two most critical boundaries.

Cohesion (High / Low)

How related the responsibilities within a single module are. High cohesion: ai-provider.doctor.ts contains only Doctor-related AI logic. Low cohesion: App.tsx contains layout, export, diagnostic orchestration, and command palette definition — four unrelated concerns in one file.

Interface Segregation Principle (ISP)

No client should be forced to depend on methods it does not use. When the Gist workspace imports AIProvider, it depends on all 35 methods, including auditSourceUsage and generateSprintPlan which it will never call. Splitting the interface into focused capabilities would satisfy ISP.

Accidental Complexity (Brooks)

Complexity that arises from the solution's implementation rather than the problem itself. The problem of analyzing academic prose is essentially complex. The problem of 26 boolean modal flags is accidentally complex — it could be modeled differently. Not all accidental complexity is worth removing (the booleans work), but knowing which kind you're looking at changes your response to it.

Sedimentary Development

A pattern characteristic of AI-assisted development: each session deposits a new layer of features atop the existing structure, preserving the shape of what came before. Like geological sediment, each layer is internally consistent, but the aggregate reveals strata of different eras — modal framing from an earlier aesthetic, import paths from a deprecated structure, utility functions that predate the shared library. This is not a defect; it is the natural residue of rapid iteration. But periodic consolidation (the architectural equivalent of metamorphism) keeps it from becoming brittle.

Protective Belt (Lakatos)

The set of auxiliary hypotheses that surround and protect the hard core of a research programme. In TreemapWriter2, the features (workspaces, modals, AI flows) are the protective belt: they can be added, modified, or removed without threatening the domain types and architectural seams that form the core. When a new feature proposal would require collapsing a domain type (e.g., merging SectionSpec with SectionAnalysis), that is a signal that the proposal is fighting the core — and the core should win.

Agent-Legibility

A first-class quality attribute of this codebase: the property that an AI agent (or a returning human) can understand how to extend the system correctly without re-deriving design intent. The documentation set (VISION/AGENTS/STATUS/migration-log) is the primary agent-legibility mechanism. Every session that pays the "re-derivation tax" — an agent spending tokens to figure out what another agent already figured out — is a session the user paid for twice.

Feature Envy

When code in module A keeps reaching into module B's data. In TreemapWriter2, the project-state slice reaches into nearly every other slice via get().otherSliceAction(). This is a controlled feature envy — centralized in one orchestrator rather than scattered — but it is still the reason project-state.ts is 689 lines.

§Coda: The Historiographer's Verdict

TreemapWriter2 is a system in the condition that Thomas Kuhn would call mature normal science. The paradigm — the domain types, the layered architecture, the source-of-truth hierarchy, the dependency-inversion seams — is settled and productive. The work being done within that paradigm (shipping workspaces, adding AI flows, refining the writing experience) is puzzle-solving of a high order. The anomalies that have accumulated (the growing god-interface, the flat modals directory, the residual App.tsx handlers) are not symptoms of a paradigm in crisis; they are the natural friction of a productive paradigm that has not yet been subjected to a round of consolidation.

The historiography of this codebase is unusually legible. The migration-log is an honest, dated record of every major change. The FOUNDING.md preserves the original mandate. VISION.md preserves the principles. AGENTS.md operationalizes them. This is a system that knows its own history, which is rare. The challenge is not that the history is lost, but that the cumulative weight of the history has made the present harder to navigate.

The clear wins identified in Section VII are not a paradigm shift. They are the consolidation that a productive paradigm periodically needs: tightening the interfaces, organizing the file system, extracting the shared primitives, completing the interrupted migrations. The domain types should not be touched. The dependency-inversion seams should not be touched. The source-of-truth hierarchy should not be touched. These are the hard core. Everything else — the protective belt — can be reorganized, consolidated, and clarified without threatening the theoretical commitments that make this system distinctive.

The deepest architectural truth about TreemapWriter2 is that its domain types already encode a serious theory of academic prose, and for too long they were jostling for room inside a JSON blob, next to sidebarWidth and showProjectModal. The refactor that produced today's architecture existed to give that philosophical model the architectural seat it had earned. The consolidation work ahead exists to keep that seat clear.
